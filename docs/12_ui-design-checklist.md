# 12_ui-design-checklist.md — Quiet SaaS デザイン・チェックリスト

対象: `apps/web`

---

## 0. 前提ルール（これが揃うと「気持ち良い」が勝手に出る）

### 0-1. デザイントークン（必須）

- [ ] 色は「意味」で決まっている（装飾ではなく役割）
  - [ ] `background` / `foreground`
  - [ ] `muted` / `muted-foreground`
  - [ ] `border` / `input` / `ring`
  - [ ] `primary` / `primary-foreground`
  - [ ] `secondary` / `secondary-foreground`
  - [ ] `destructive` / `destructive-foreground`
  - [ ] `accent` / `accent-foreground`
- [ ] 角丸は1〜2種類に制限されている（例: `--radius` を基準）
- [ ] 影の段数は少ない（0〜2段）。多段シャドウは禁止寄り
- [ ] 線（border）の太さ・濃さが統一されている（1px基準）
- [ ] 文字サイズの段階が増えすぎていない
- [ ] 余白スケールは8px系に寄せる（2/3/4/6/8/10/12中心）
- [ ] 「例外」に理由がある（密度要件など）

### 0-2. Quiet SaaSの強弱ルール

- [ ] 1画面で一番強い要素は1つだけ（主ボタン or 主カード or 主メッセージ）
- [ ] 強調は最大2要素まで（太字/サイズ/色/背景/枠/アイコンを同時多用しない）
- [ ] 情報は要点を手前、詳細を奥に置く（折りたたみ/詳細画面/tooltip）

---

## 1. レイアウト（骨格）

### 1-1. 画面のグリッドと幅

- [ ] 画面の最大幅が統一されている（例: `max-w-5xl`）
- [ ] 左右余白がデバイスごとに自然（例: `px-4 sm:px-6 lg:px-8`）
- [ ] フル幅が必要な画面が定義されている（例: カレンダー/大型テーブル）
- [ ] セクション間の縦リズムが一定（例: `space-y-6`）

### 1-2. 整列（alignment）

- [ ] 見出し/説明/ボタンのベースラインが揃っている
- [ ] フォームのラベル/入力/補足/エラーが一直線
- [ ] アイコンサイズとテキストサイズが釣り合っている
- [ ] カード内余白（上下左右）が統一されている

---

## 2. 余白（Spacing）— Quietの最重要

### 2-1. リズム（vertical rhythm）

- [ ] 画面の `space-y-*` ルールが統一されている
- [ ] 見出しの上下余白が毎回同じ
- [ ] 入力同士の間隔が均一

### 2-2. 密度のポリシー

- [ ] 密度は「通常」と「高密度（一覧・テーブル）」の2種類
- [ ] 一覧だけ `py-2`、通常は `py-3/4` など差が意図的
- [ ] 高密度画面ほど区切り（divider）ルールが明確

---

## 3. タイポグラフィ（読み心地）

### 3-1. 階層（hierarchy）

- [ ] ページタイトル/セクション見出し/本文/補足の役割が固定
- [ ] 同一役割のサイズ/太さにブレがない
- [ ] 本文の行間が読みやすい（目安: `leading-6`）
- [ ] 数字が多い画面で桁・単位・区切りを整えている

### 3-2. 文章（microcopy）

- [ ] ボタン文言は動詞で統一（作成/保存/更新/公開）
- [ ] ラベルとプレースホルダーの役割が分離
- [ ] エラー文言が具体
- [ ] 空状態で次の行動が示される

---

## 4. 色・コントラスト（Quietの品）

### 4-1. 色の使い分け

- [ ] 色は意味で使っている（注意/成功/危険/情報）
- [ ] 装飾目的の色が増えていない
- [ ] `muted` を多用しすぎていない
- [ ] CTA優先度が明確（`primary > secondary > ghost/link`）

### 4-2. コントラスト・可読性

- [ ] 主要テキストのコントラストが十分
- [ ] `muted-foreground` が読める濃さ
- [ ] `disabled` が消えすぎない
- [ ] `destructive` の使い過ぎがない

### 4-3. ダークモード（対応する場合）

- [ ] 境界線/入力枠/hover が破綻しない
- [ ] 影より境界/背景差で階層を作れている
- [ ] `destructive` が強すぎない

---

## 5. コンポーネント一貫性（shadcnの強み）

### 5-1. Button

- [ ] 主ボタンは画面単位で1つ
- [ ] 破壊的操作は `destructive` で統一
- [ ] サイズ乱立がない（用途固定）
- [ ] アイコン付きボタンの余白が一定
- [ ] `disabled/loading` の見え方が統一

### 5-2. Input / Select / Textarea

- [ ] ラベルがある（placeholderだけにしない）
- [ ] 必須/任意が明確
- [ ] エラー表示位置が統一
- [ ] help text と error の優先順位が明確
- [ ] Select/Combobox の空状態・検索・0件が丁寧

### 5-3. Card / Section

- [ ] カード余白が統一
- [ ] 見出し/説明/アクション配置が一貫
- [ ] 情報カードと操作カードが混ざっていない

### 5-4. Table / List

- [ ] 行の高さが一定
- [ ] 列の整列が明確（数字は右寄せ等）
- [ ] hoverが主張しすぎない
- [ ] selected と hover が区別できる
- [ ] 空状態/フィルター0件が自然

### 5-5. Dialog / Drawer / Popover / Tooltip

- [ ] オーバーレイ濃度が適切
- [ ] 閉じる操作が複数ある（×/ESC/外側クリック等）
- [ ] 用途が混ざっていない（重い確認は Dialog など）
- [ ] スクロール時挙動が破綻しない

### 5-6. Toast / Alert / Banner

- [ ] 成功/失敗/注意の見え方が統一
- [ ] 次アクションが入っている（再試行/詳細/戻る）
- [ ] 連打でトースト地獄にならない

---

## 6. 状態設計（Quietは状態が丁寧）

### 6-1. 画面状態

- [ ] Loading: スケルトン/代替表示がある
- [ ] Empty: 理由 + 次アクションがある
- [ ] Error: 復帰導線がある（再試行/戻る等）
- [ ] Partial data: 一部取得時でも表示破綻しない

### 6-2. 操作状態

- [ ] 送信中に二重送信できない
- [ ] 保存成功は静かに確実に伝える
- [ ] 失敗時に入力内容が消えない
- [ ] 破壊的操作に `undo/confirm/soft delete` 方針がある

---

## 7. アニメーション / モーション（Quiet SaaS仕様）

ポリシー: 意味がある動きだけ、短く、小さく、揺らさない

### 7-1. モーショントークン

- [ ] duration が役割ごとに固定（例: 150/200/300ms）
  - [ ] hover/focus: 150ms前後
  - [ ] popover/dropdown: 200ms前後
  - [ ] dialog/sheet: 250–300ms前後
- [ ] easing が統一されている
- [ ] 移動量が小さい（2〜6px）
- [ ] 拡縮は控えめ（95%→100%程度）
- [ ] `transition-all` を乱用していない

### 7-2. 動きを入れる場所（許可ゾーン）

- [ ] 出現/消失（Dialog/Popover/Dropdown/Toast）
- [ ] 状態変化（hover/pressed/focus/selected）
- [ ] 処理中（skeleton/subtle spinner）
- [ ] 装飾目的の常時アニメは原則入れない

### 7-3. パフォーマンス

- [ ] `transform/opacity` 中心で動かす
- [ ] 高さアニメは最小限
- [ ] `box-shadow` アニメは控えめ
- [ ] 同時に複数箇所を動かしすぎない（最大1〜2箇所）

### 7-4. アクセシビリティ（動きが苦手な人）

- [ ] `prefers-reduced-motion` に対応
  - [ ] `motion-reduce:animate-none`
  - [ ] `motion-reduce:transition-none`
- [ ] モーションがなくても理解できる

### 7-5. コンポーネント別期待値

- [ ] Dialog: overlayはフェード、コンテンツは小さなフェード/移動/拡縮
- [ ] Popover/Dropdown: フェード + 小移動（方向一貫）
- [ ] Toast: 出現短く、滞在邪魔せず、消失自然
- [ ] Collapse/Accordion: 情報量に応じて開閉速度を調整

---

## 8. アクセシビリティ（Quiet＝優しさ）

### 8-1. フォーカスと操作

- [ ] Tab移動で全操作可能
- [ ] `focus-visible` が見える
- [ ] フォーカス順が自然（モーダル内トラップ含む）
- [ ] クリック領域が狭すぎない

### 8-2. ラベルと説明

- [ ] 入力に label がある
- [ ] エラーが入力と結びついている
- [ ] アイコンだけのボタンに説明がある（`aria-label`等）

### 8-3. 色依存の回避

- [ ] 成功/失敗が色以外でも分かる（アイコン/文言/形）

---

## 9. レスポンシブ（崩れた瞬間に信頼が落ちる）

- [ ] スマホで操作可能（タップ領域、スクロール、固定ヘッダ干渉なし）
- [ ] 重要情報が折り返しで欠けない（一覧/日付/時間）
- [ ] 1カラム化ルールがある（カード縦積み、アクション下配置等）
- [ ] モーダル/ポップオーバーが画面外に出ない

---

## 10. 一覧・フォームの業務UI品質

### 10-1. 一覧

- [ ] フィルター/検索配置が一貫
- [ ] ソート表現が分かりやすい
- [ ] 行クリックと行内ボタンが競合しない
- [ ] ページング/無限スクロール方針が統一

### 10-2. フォーム

- [ ] 入力順が自然（上→下）
- [ ] 保存ボタン位置が一貫
- [ ] 破棄（キャンセル）導線が明確
- [ ] 未保存変更の検知方針がある

---

## 11. 仕上げ（Quietの品が出るところ）

- [ ] 文言トーンが一貫（敬体/常体）
- [ ] 表記ゆれがない（用語、単位、日付フォーマット）
- [ ] 過剰装飾がない（線/影/色の増やしすぎ回避）
- [ ] 「怖い瞬間」がない（突然消える、保存可否不明、戻れない）
